## Slide 1：封面

各位老师、同学好，我们小组带来的是单细胞高维向量近似最近邻检索系统。项目以肝脏单细胞测序数据为场景，搭建了一个从数据上传、索引构建、相似检索到自然语言问答的完整 Web 平台，技术栈覆盖前端、后端、向量引擎和大模型，整个系统已经跑通了真实数据。下面正式开始今天的汇报。

## Slide 2：目录

汇报分六个部分。第一部分讲单细胞数据为什么需要 ANN；第二部分对照课程要求梳理我们做了哪些功能；第三和第四部分介绍系统设计与核心实现；第五部分展开三个加分项；最后一部分给出真实数据上的性能实测和交付清单。整体大约十分钟，欢迎随时打断提问。

## Slide 3：项目背景

单细胞测序一次实验就能产出十万级别的细胞，每个细胞用基因表达向量来表示，维度通常在三十到几百之间。我们用的肝脏数据有六万九千零三十二个细胞。这种规模下，精确 KNN 查询动辄几百毫秒，根本撑不起前端的交互。ANN 算法可以在召回率几乎不掉的前提下，把单次查询压到微秒级，这就是我们项目的立项动机。

## Slide 4：需求分析

课程要求的五大模块我们全部实现：用户认证、数据管理、索引构建、查询检索、可视化。账号用 JWT 鉴权，还支持管理员角色。在这个基础上我们把三个加分项全部做到了：第一个是多数据集联合检索，第二个是自适应 HNSW 算法改进，第三个是基于大模型的 RAG 自然语言问答。后面会逐项展开。

## Slide 5：系统架构总览

整体采用经典的前后端分离加任务队列架构。浏览器端是 React 单页应用，通过 Nginx 反向代理访问 FastAPI 网关。耗时操作，比如索引构建，由 ARQ Worker 从 Redis 队列里取任务异步处理。向量引擎封装在独立的服务层，五种后端可以热切换。元数据存在 PostgreSQL，向量文件和索引落在本地文件系统。整个栈用 Docker Compose 一键启动。

## Slide 6：技术栈

后端选了 Python 三点十二加 FastAPI，纯异步路由，配合 SQLAlchemy 二代的 async 引擎。前端用 React 十九加 TypeScript，可视化交给 Plotly。向量引擎方面，FAISS 和 HNSWLIB 同时引入，scikit-learn 提供精确检索作为基准。大模型支持通义千问和 OpenAI 兼容协议，没有外网时自动退到 Mock。基础设施加了 GitHub Actions 和 pre-commit，保证代码质量。

## Slide 7：ANN 引擎抽象

我们把所有 ANN 库都抽象成一个统一接口 IndexBackend，对外暴露构建、检索、保存、加载、内存估计这几个方法。然后通过工厂函数根据名字创建具体后端，再用一个 LRU 缓存把已加载的索引常驻内存。这样上层路由完全不感知底层用的是 hnswlib 还是 faiss，新增算法只要写一个子类就行。我们的自适应算法就是这样接入的。

## Slide 8：核心代码

左边是抽象基类，五个抽象方法定义了一个完整的 ANN 后端契约。右边是自适应算法的核心循环：每一轮检索后判定是否稳定，首轮看相对距离间隔，后续轮看 Top-K 集合重合度。稳定的查询直接返回，没稳定的把 ef 翻倍重查。这种按查询粒度的细粒度控制，是普通固定 ef 的算法做不到的，也是我们改进的关键点。

## Slide 9：数据模型

数据库一共四张表。用户表负责认证和角色；数据集表记录上传的 h5ad 文件、预处理后的向量路径、维度、状态；索引记录表保存每个数据集上构建的多个索引，包括后端、参数、构建耗时、内存占用；最后是检索日志表，每次查询的延迟、过滤条件都会落库，方便后续做性能分析。所有表的 schema 变更都用 Alembic 管理。

## Slide 10：检索流水线

一次检索的完整链路是这样的：用户在前端选好数据集和索引提交查询；FastAPI 校验权限、查出索引记录；IndexCache 命中就直接拿后端实例，否则从磁盘加载。真正的 numpy 计算用 to_thread 卸载到线程池，不阻塞事件循环。后端返回原始索引和距离，再做 metadata 过滤、剔除自身、排名重排，最后写检索日志、返回 Top-K。前端拿到结果同时刷新表格和 UMAP 高亮。

## Slide 11：加分项一 多数据集联合

实际科研场景里，一个研究往往涉及多个数据集，比如不同人群的肝脏样本。我们提供了 multi-dataset 接口，传入一组 dataset_id，后端用 asyncio.gather 并发查询所有数据集，然后对每个数据集的距离做 min-max 归一化，让它们落到同一个相对尺度，再合并排序取全局 Top-K。这样跨数据集结果的可比性就有了，整体延迟约等于最慢那个数据集，而不是相加。

## Slide 12：加分项二 自适应 HNSW

标准 HNSW 的 ef_search 参数是全局固定的，但实际查询难度差异很大：有的查询很容易，几乎不用扩散就能找到；有的很难，ef 设小了召回不够。我们的策略是起始 ef 设小，多取一些候选；首轮用相对距离间隔判稳，后续轮用 Top-K 集合重合度判稳；不稳定的查询独立升档重查，最大不超过五百一十二。实测平均 ef 从六十四降到五十点六，召回率几乎没变，平均算力省了百分之二十一。

## Slide 13：加分项三 RAG 自然语言

RAG 模块的设计核心是 parse、search、summarize 三步。用户输入自然语言，比如"找二十个像肝细胞的内皮细胞"，LLM 先解析成结构化参数：过滤条件 cell_type 等于 endothelial、Top-K 等于二十；然后用过滤命中的第一条向量做相似检索；最后再让 LLM 对结果生成中文总结。三个客户端协议化设计，DashScope、OpenAI、Mock 可切换，没外网时也能完整演示。

## Slide 14：实测数据

我们在三万条肝脏 PCA 向量上跑了基准。构建耗时方面 hnswlib 是零点二二四秒，faiss 系列零点一八到零点二五秒之间。单线程 Top-10 检索 hnswlib 的 p50 只有零点零一六毫秒，p99 也才零点零二一毫秒，QPS 超过六万三。Recall@10 高达零点九九九六。IVF-PQ 内存只有零点二九兆，但召回掉到零点八，适合冷数据。自适应版本在三种权衡里取得了不错的平衡。

## Slide 16：质量保障

质量层面，后端 pytest 三十五个用例全部通过，覆盖认证、数据集、ANN、检索、评测、RAG、健康检查七个模块。代码层面 ruff、mypy、ESLint、Prettier、TypeScript 严格模式全绿；pre-commit 在提交前自动校验。CI 在 GitHub Actions 上跑 lint、test、build 三步流水线。我们一共有二十五次约定式提交，每一次大变更都对应一个完整的功能闭环。

## Slide 18：v1.1 性能优化总览

v1.0 之后我们又推进了二十六次提交，落地十一个 feat 和 perf 任务，全部已合入主分支并配套测试。这里用一张表把改动分成六个模块讲清楚：缓存层加了 IndexCache 启动预热和 Redis 检索缓存；索引存储引入 mmap 加载和 float16 落盘；算法侧用 numba 把 BruteBackend 单线程加速三点一五倍；接口层新增批量、SSE 流式、Anthropic RAG；数据库消除了 N 加 1；前端做了 plotly 瘦身、移动响应式和 skeleton。每一项都带数字，下面三张依次展开。

## Slide 19：双层缓存设计

我们做了两级缓存：L1 是进程内的 IndexCache，LRU 容量四，按 index_id 常驻已加载后端，配合 asyncio.Lock 防止重复加载；F4 在 lifespan 里预热最近三个 ready 索引，彻底消掉首查冷启动毛刺。L2 是跨进程的 Redis SearchCache，key 用 SHA256 拼 index_id、top_k、向量哈希和过滤条件，命中直接返 cache_hit 等于 true，Redis 不可用时优雅降级为透传。两级 stats 拼到同一个监控接口暴露给前端。

## Slide 20：算法与数据加速

这一张是性能优化的三件套。F3 mmap 让 numpy 按页惰性读盘，大底库冷启动内存对半砍。F5 float16 落盘消融实测：磁盘减半，hnswlib Recall@10 只从零点九五六掉到零点九五二，p95 和 QPS 都在测量噪声内，性价比极高，还能和 mmap 正交叠加。P2 给 BruteBackend 加了 numba njit 并行内核，五万乘五十的数据，单线程 p50 从一点三四毫秒降到零点四二六毫秒，三点一五倍加速；并发场景非阻塞锁自动降级到 BLAS 路径，规避 workqueue 崩溃。

## Slide 21：API 扩展矩阵

v1.1 在检索接口上扩成四种姿势。F1 batch 接口一次最多五十条查询，asyncio gather 并发跑，每条独立走 Redis 缓存，适合评测脚本一把梭。F6 SSE 流式接口把 Top-K 拆成 event hit 一条条推，每条间隔二十毫秒，前端能逐条飞入，体感延迟显著降低，末尾 event done 给汇总指标。F7 多数据集 ensemble 是 v1.0 加分项，做了 min-max 归一化跨集排序。F8 新增 Anthropic Claude Opus 四客户端，RAG 多一种选择，失败自动 fallback Mock，接口文档同步从二十一个涨到三十一个。

## Slide 22：总结

总结一下，我们交付了一个完整的端到端 Web 系统，从注册上传一路打通到自然语言问答；三个加分项全部落地；性能上做到微秒级延迟、九十九点九六召回率、六万 QPS 的可量产水平。v1.1 又叠加了二十六次提交的优化，缓存、索引存储、算法、API、前端全链路都有量化收益。整个项目过程中我们也学到了不少东西，包括异步框架、向量引擎、LLM 集成、端到端测试。汇报到此结束，欢迎老师同学提问。谢谢大家。
