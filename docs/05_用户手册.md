# 五、用户手册

## 5.1 系统介绍

**单细胞高维向量近似最近邻 (ANN) 检索系统** 是一个面向生物信息学研究者的 Web 应用，支持上传 `.h5ad` 单细胞数据、构建多种 ANN 索引、执行高速相似细胞检索、跨数据集联合分析与可视化展示，并提供 RAG 自然语言查询入口与算法基准评测。

### 1.1.1 主要能力

- **多后端 ANN**：Brute、HNSWLIB、FAISS-HNSW、FAISS-IVFPQ、Adaptive-HNSW 五种引擎，参数面板可视化调参；
- **条件过滤检索**：基于 `obs` 元信息（如 `cell_type`、`disease`）的精准过滤；
- **多数据集联合检索**：并发检索 + 距离归一化合并，结果带来源标记；
- **RAG 自然语言查询**：自然语言 → 结构化检索参数 → ANN → 总结；
- **可视化**：UMAP 二维投影 + Top-K 高亮 + 距离直方图；
- **性能评测**：与 Brute-force ground truth 对比的 Recall / Latency / QPS；
- **异步任务队列**：长耗时任务（预处理、索引构建、评测）走 ARQ + Redis，前端轮询状态。

### 1.1.2 适用人群

- 生物信息学研究者：在大规模单细胞数据中寻找相似细胞、注释亚群；
- 算法工程师：横向对比不同 ANN 后端在真实生物数据上的表现；
- 课程评审 / 答辩观众：通过 Web 界面快速理解端到端系统能力。

## 5.2 部署指南

系统提供两套部署方式：**Docker Compose 一键启动**（推荐，零环境冲突）与**本地原生开发模式**（适合二次开发）。

### 5.2.1 前置要求

| 项 | 要求 |
| --- | --- |
| CPU / 内存 | 4 核 8 GB 起步；演示推荐 8 核 16 GB |
| 磁盘 | 至少 10 GB 可用空间（`liver.h5ad` 约 1.4 GB）|
| 操作系统 | macOS 14+ / Ubuntu 22.04+ / Windows 11 + WSL2 |
| Docker | Docker Desktop / Engine 26+，Docker Compose v2 |
| 可选（本地开发）| Python 3.12、Node 22、uv、pnpm |

### 5.2.2 Docker Compose 一键启动（推荐）

```bash
git clone <repo-url>
cd ann_search

cp .env.example .env                # 按需修改 SECRET_KEY / LLM_API_KEY
make up                             # 启动 postgres + redis + backend + worker + frontend
make migrate                        # 首次启动需要应用数据库迁移
make logs                           # 实时查看各容器日志
```

启动完成后访问：

- 前端 UI: <http://localhost:5173>
- 后端 Swagger UI: <http://localhost:8000/docs>
- 后端 ReDoc: <http://localhost:8000/redoc>
- OpenAPI JSON: <http://localhost:8000/openapi.json>
- PostgreSQL: `localhost:5432`（账号 `ann` / 密码 `ann`）
- Redis: `localhost:6379`

### 5.2.3 常用 Make 命令

```bash
make help            # 列出全部命令
```

### 5.2.4 本地原生开发模式

适合需要单步调试或在容器外跑性能基准的场景。

**后端**：

```bash
cd backend
uv sync                                                  # 安装 Python 依赖（含 dev 组）
docker compose -f ../infra/docker-compose.yml up -d postgres redis
uv run alembic upgrade head                             # 应用迁移
uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

**ARQ Worker**（在另一终端）：

```bash
cd backend
uv run arq app.tasks.worker.WorkerSettings
```

**前端**：

```bash
cd frontend
pnpm install            # 或 npm install
pnpm dev                # 默认 http://localhost:5173
```

## 5.5 索引构建

### 5.5.1 操作流程

1. 在「数据集」页面点击目标行的「构建索引」，或进入「索引管理」页面；
2. 在右侧表单选择参数：
   - **数据集**：从下拉框选择已就绪的数据集；
   - **后端**：`hnswlib` / `faiss-hnsw` / `faiss-ivfpq` / `brute` / `adaptive-hnsw`；
   - **距离度量**：`l2` / `cosine` / `ip`；
   - **后端参数**（按 backend 联动渲染）：
     - **HNSWLIB / FAISS-HNSW / Adaptive-HNSW**：`M`（默认 16）、`ef_construction`（默认 200）、`ef_search`（默认 64）；
     - **FAISS-IVFPQ**：`nlist`（建议 ≈ √N）、`m`（子向量数）、`nbits`（默认 8）、`nprobe`（默认 16）；
     - **Brute**：无参数。
3. 点击「开始构建」，后端立即返回 `202 Accepted` 与 ARQ `task_id`；
4. 列表中该索引出现，状态先后经过 `building → ready`；
5. 构建完成后 `build_time_seconds` 与 `memory_mb` 字段会被填充。

> 索引管理页（hnswlib 默认参数）：

![索引管理页](e2e_screenshots/05_index_page.png)

> 索引构建成功，进入 `ready`：

![索引 ready](e2e_screenshots/06_index_ready.png)

### 5.5.2 算法选择建议

| 数据规模 | 维度 | 推荐后端 | 备注 |
| --- | --- | --- | --- |
| < 1 万 | 任意 | `brute` | 精确，用于评测 ground truth |
| 1 ~ 10 万 | < 256 | `hnswlib` | 默认首选，综合最优 |
| 10 ~ 100 万 | < 256 | `hnswlib` / `faiss-hnsw` | 内存敏感选 HNSWLIB，需要并行选 FAISS |
| > 100 万 / 内存受限 | 任意 | `faiss-ivfpq` | 量化压缩，建议叠加 brute 重排序 |
| query 难度分布不均 | 任意 | `adaptive-hnsw` | 动态 ef，兼顾尾部召回与平均延迟 |

## 5.6 相似检索

### 5.6.1 按 cell_id 检索

1. 进入「检索」页面；
2. 选择数据集与索引；
3. 在「按细胞 ID」Tab 输入一个已存在的 `cell_id`；
4. 设置 `Top-K`（默认 10）；
5. 可选：展开「过滤条件」按 `obs` 列添加；
6. 点击「检索」。

> 检索结果（命中 10 条，含距离与 cell_type）：

![检索结果](e2e_screenshots/07_search_result.png)

### 5.6.2 按自定义向量检索

切到「按向量」Tab：

- 直接粘贴 JSON 数组（与数据集 `vector_dim` 长度一致），或
- 上传 `.npy` / `.json` 文件（前端自动解析）；

如果维度与数据集不一致，会得到 `422`：

```
向量维度 32 与数据集维度 30 不一致
```

### 5.6.3 条件过滤

在「过滤条件」面板中，按 `obs` 元信息列添加：

- **等值**：`cell_type = Hepatocyte`；
- **多值（IN）**：`disease IN [NAFLD, Healthy]`；
- **范围（数值）**：`AgeYears BETWEEN 0 AND 18`。

系统采用 **post-filter** 策略：先在 ANN 层取 `top_k × 5` 候选，再按 metadata 过滤截取 Top-K，保证近似召回不被 pre-filter 严重伤害。

### 5.6.4 多数据集联合检索（扩展）

1. 切到「多数据集」Tab；
2. 勾选多个 `dataset_id`（与对应索引）；
3. 选择查询源：
   - 「以已有 cell_id 为查询点」：选其中一个数据集作为 `source_dataset_id`；
   - 「以自定义向量为查询点」；
4. 系统会并发检索每个数据集，按 min-max 归一化距离合并 Top-K；
5. 每条结果带 `source_dataset_id`，便于定位来源。

### 5.6.5 检索结果说明

每条命中包含以下字段：

| 字段 | 含义 |
| --- | --- |
| `rank` | 结果排名（从 1 开始）|
| `cell_id` | 细胞 ID（`obs.index`）|
| `distance` | 与查询点的距离（L2 / 1-cosine / -IP）|
| `meta` | `obs` 元信息（按 `meta_columns` 截取）|
| `source_dataset_id` | 仅多数据集检索时填充 |

页面响应头还会显示 `latency_ms` / `total_candidates` / `index_backend`，方便对比不同后端性能。

## 5.8 RAG 自然语言查询（扩展）

### 5.8.1 操作流程

1. 进入「RAG 对话」页面；
2. 选择数据集与索引；
3. 在输入框输入自然语言提问，例如：
   - "找出与肝细胞类似的细胞 Top-10"；
   - "在儿童肝脏数据中检索与 cell_id `liver_pediatric_0001` 类似的 NAFLD 样本"；
   - "列出 30 个高表达 ALB 的肝细胞"；
4. 系统会：
   - 由 LLM（默认 `mock` provider）将自然语言解析为结构化检索参数 (`parsed`)：`cell_id` / `filters` / `top_k`；
   - 调用 ANN 后端执行结构化检索；
   - 由 LLM 生成自然语言总结（如细胞类型分布、突出统计）；
5. 页面同时展示 `parsed` 参数、命中条目表格与最终 `answer` 段落。

> RAG 对话页：

![RAG 自然语言对话](e2e_screenshots/09_rag.png)

### 5.8.2 LLM Provider 切换

默认 `LLM_PROVIDER=mock` 走关键词规则与模板化生成，**无需任何外部 API Key 即可工作**。如需更强能力，可在 `.env` 中切换：

```dotenv
LLM_PROVIDER=openai
LLM_API_KEY=sk-...
LLM_MODEL=gpt-4o-mini
```

> 切换后请确认数据隐私合规：RAG 仅会向 LLM 发送结构化参数与匿名 `cell_id` 列表，不会上传完整 `obs` 元数据。

## 5.9 常见问题排查 (FAQ)

### Q1. 上传 `.h5ad` 后一直 `preprocessing`，没变成 `ready`？

**可能原因**：

- 文件较大，Scanpy 解析需要时间（1.3 GB liver.h5ad 实测 ~85 s）；
- Redis / ARQ Worker 未启动；
- 文件不是合法 AnnData 容器。

**排查步骤**：

```bash
make logs                       # 看 worker 日志
docker compose ps               # 看 worker 容器状态
make backend-shell
uv run arq app.tasks.worker.WorkerSettings  # 手动启动 worker
```

如果 worker 报错 `Cannot open file ...`，请确认 `DATA_DIR` 是否挂载正确。

### Q2. 索引构建失败 `status=failed`？

常见原因：

- 数据集尚未 `ready` 即触发构建（前端已禁用，但 API 直调可能遇到）；
- IVF-PQ `nlist` 大于样本数（自动建议 `nlist ≈ √N`，不要手动设过大值）；
- 内存不足（超大规模数据需要切到 IVF-PQ 节省内存）；
- 维度过低（< 8 维不适合 PQ 量化）。

通过 `GET /api/v1/indexes/{id}` 查看 `params` 与服务端日志可定位具体原因。

### Q3. 检索返回 `409 索引尚未 ready`？

索引仍在 `building` 状态。在「索引管理」页等待状态轮询变为 `ready` 再发起检索。如长时间不变，参考 Q2 排查 worker。

### Q4. 检索结果与 Brute-force 对比明显不一致？

ANN 算法在召回率上本就 < 100%。可：

- 在评测页查看具体 `Recall@K`；
- 适当增大 `efSearch`（HNSW）或 `nprobe`（IVF-PQ）；
- 切换到 `hnswlib` / `adaptive-hnsw` 后端（Recall@10 通常 > 0.999）。

### Q5. 前端登录后立刻被踢回登录页？

通常是 `SECRET_KEY` 在 `.env` 中改过且后端容器未重启，导致旧 token 验签失败。执行：

```bash
make down
make up
```

清空浏览器 `localStorage` 后重新登录即可。

### Q6. RAG 页面输出"未能解析"？

`LLM_PROVIDER=mock` 模式下仅识别关键词（如 `cell_type` 取值、`cell_id` 模式）。如查询过于宽泛，可：

- 改写为更明确的自然语言，包含具体细胞类型或 cell_id；
- 切换到 `openai` 提供更强解析能力（需 API Key）。

### Q7. 想清空所有数据从零开始？

```bash
make down
docker volume rm ann_search_pgdata ann_search_redisdata   # 卷名按实际为准
rm -rf data/* indexes/*
make up
make migrate
```

> ⚠️ 该操作会清空全部用户、数据集与索引，无法恢复。

### Q8. 修改环境变量后未生效？

修改 `.env` 后需重启容器（FastAPI / Worker 仅在启动时读取一次）：

```bash
make down && make up
```

### Q9. 普通用户登录后看不到「管理员」入口？

**原因**：v1.1.0 起新增 `/admin/users` 页面与一组 admin 接口，**仅 `is_admin=true` 的账号可访问**，普通账号导航栏不会展示该入口、后端也会返回 `403`。

**解决**：

1. 默认注册的账号都是普通用户。要把一个账号升级为管理员，请在容器内执行：

```bash
make backend-shell
uv run python -c "
import asyncio
from sqlalchemy import update
from app.db.session import AsyncSessionLocal
from app.models.user import User

async def main():
    async with AsyncSessionLocal() as s:
        await s.execute(update(User).where(User.username == 'demo').values(is_admin=True))
        await s.commit()
asyncio.run(main())
"
```

2. 重新登录即可在导航栏看到「管理员」入口（功能：用户 CRUD / 重置密码 / 启停账号）。
3. 也可以直接通过 `POST /api/v1/admin/users` 接口创建管理员账号，参数 `is_admin=true`。

### Q10. 想用 SSE 流式检索（F6），但浏览器请求一直挂起？

**原因**：`POST /api/v1/search/stream` 返回 `Content-Type: text/event-stream`，**必须用 `EventSource` 或 fetch 流读取**，普通 axios JSON 请求会等所有事件结束才一次性解析，体验和普通接口一样。

**解决**：

- 前端：使用 [`frontend/src/api/search.ts`](../frontend/src/api/search.ts) 的 `streamSearchByVector` 方法（已封装 fetch + ReadableStream + EventSource 解析），SearchPage 「流式」Tab 可直接观察到逐条结果。
- curl 验证：

  ```bash
  curl -N -H "Authorization: Bearer $TOKEN" \
       -H "Content-Type: application/json" \
       -d '{"index_id": 1, "vector": [...], "top_k": 10}' \
       http://localhost:8000/api/v1/search/stream
  ```

- 如果 Nginx 代理把流式缓冲了，确认 `infra/nginx/nginx.conf` 里对 `/api/v1/search/stream` 关闭了 `proxy_buffering`。

### Q11. ensemble 多后端融合（F7）的权重怎么调？

**原因**：`POST /api/v1/search/ensemble` 接受一组 `index_ids` 与可选 `weights`，按 z-score 归一化每个后端的距离再加权融合。默认权重为均匀分布（每个后端权重相同）。

**解决**：

- **不传 weights**：所有后端等权融合，适合"我想要召回最稳，不偏向任一后端"。
- **传 weights**：长度需与 `index_ids` 一致，权重越大该后端贡献越大。
  - 例如想以 hnswlib 为主、brute 用于"兜底高难 query"：`{"weights": [0.7, 0.3]}`；
  - 或想看 IVF-PQ 与 HNSW 的差异：分别试 `[1, 0]` 与 `[0, 1]`。
- **配套调参**：建议 `oversample` 取 2~5（每个后端取 `top_k * oversample` 候选再融合），过大会浪费算力。
- 详细 schema 与示例见 [`docs/06_API接口文档.md`](06_API接口文档.md#ensemble-多后端融合检索)。

### Q12. SSE 流式跟普通检索有什么区别？

**核心区别**：普通 `POST /api/v1/search/by-vector` 算完一次性返回 JSON，前端必须等所有结果到齐才能渲染；SSE `POST /api/v1/search/by-vector-stream` 算完后逐条以 `event: hit` 推送，前端可"边收边渲染"，最后一条 `event: done` 携带 `latency_ms / total_candidates` 汇总信息。

**怎么选**：

- **Top-K ≤ 50 且后端快（hnswlib）**：用普通接口即可，体验差异不明显；
- **Top-K ≥ 100 或后端慢（brute / IVF-PQ + 大 nprobe）**：用 SSE，显著改善"白屏"等待感；
- **脚本化批处理**：用普通接口或 batch 接口（10.1），SSE 解析复杂没必要。

> 实测：SSE 总 `latency_ms` 与同参数普通接口几乎一致（±2 ms 内）；它解决的是**感知延迟**而非真实延迟。

### Q13. Ensemble 融合时如何选索引？

**经验法则**：

- **要召回稳**：选 `hnswlib + faiss-hnsw + brute`（三种不同实现，投票一致性高，`voted_by=[...]` 长度往往为 3）；
- **想看后端差异**：选 `hnswlib + faiss-ivfpq`，观察 `voted_by` 中哪些 cell 只命中一家（往往是 IVF-PQ 量化误差所致）；
- **不建议**：全选同一后端不同参数（如三个不同 `ef_search` 的 hnswlib），信息冗余、收益极低；
- **数量上限**：服务端硬限 2~5，**3 个通常是最佳折中**（端到端延迟约等于最慢一路而非求和，因为是 `asyncio.gather` 并发）。

> 输出的 `voted_by` 列出命中该 cell 的索引 ID 列表，可作为"该结果可信度"的简易指标——多个后端一致命中的 cell 比只被一家命中的更可靠。

### Q14. Anthropic 和 OpenAI / DashScope 怎么切换？

**统一通过 `.env` 的 `LLM_PROVIDER` 切换**，无需改代码：

| Provider | `LLM_PROVIDER` | `LLM_MODEL` 推荐 | Key 字段 |
| --- | --- | --- | --- |
| Mock（默认，零外部依赖） | `mock` | -（关键字规则） | 无 |
| Anthropic Claude | `anthropic` | `claude-opus-4-20250514` | `ANTHROPIC_API_KEY` 或 `LLM_API_KEY` |
| OpenAI | `openai` | `gpt-4o-mini` | `LLM_API_KEY` + 可选 `LLM_BASE_URL`（兼容 endpoint）|
| DashScope（阿里通义） | `dashscope` | `qwen-plus` | `LLM_API_KEY` |

**切换步骤**（四家通用）：

1. 编辑 `.env` 改 `LLM_PROVIDER` + 对应 `LLM_API_KEY` + `LLM_MODEL`；Anthropic 也可单独配 `ANTHROPIC_API_KEY`（优先级高于 `LLM_API_KEY`）；
2. `make down && make up` 重启容器（provider 仅在启动时读取一次）；
3. 进 `/rag` 提问，首条响应延迟 ≈ LLM 网络 RTT + ANN 时间，通常 1~3 s。

> 如果想横向对比四家，可在 `backend/scripts/` 写小脚本依次切 `LLM_PROVIDER` 重启 → 同一 query 跑一次记录 `parsed` 与 `answer`。

### Q15. 管理员页面 `/admin/users` 看不到或返回 `403`？

**前置**：菜单显示「用户管理」以及接口返回 `200` 都要求 `users.role='admin'`。Q9 已介绍提权方法，本条聚焦排查清单：

1. **看角色**：调 `GET /api/v1/auth/me` 或解码 `localStorage.ann_search_token` payload，确认 `role == 'admin'`；若仍是 `user`，先按 Q9 提权；
2. **看缓存**：提权后必须**重新登录**让前端拿到新 JWT；直接刷新页面通常无效（旧 token 里仍是 `user`，前端 `useAuthStore` 据此判断是否注入菜单）；
3. **看后端**：直接 curl `GET /api/v1/admin/users` 返 `403`，多半是 JWT 过期或 `SECRET_KEY` 不一致（参考 Q5）；返 `401` 说明请求未带 `Authorization: Bearer`；
4. **看 Sider 渲染**：源码见 [`frontend/src/components/Layout.tsx`](../frontend/src/components/Layout.tsx)（`user?.role === 'admin'` 决定是否注入「用户管理」菜单），若 store 里 `user` 为 `null`，先排查登录是否成功（控制台 Network → `/api/v1/auth/login`）。

## 5.10 性能调优建议

### 5.10.1 HNSW 系列参数调优

| 参数 | 含义 | 调参方向 |
| --- | --- | --- |
| `M` | 每个节点的最大邻居数 | 增大 → 召回上升、内存增大、构建变慢；典型 16~48 |
| `ef_construction` | 构建时的候选列表大小 | 增大 → 召回上升、构建变慢；典型 100~400 |
| `ef_search` | 检索时的候选列表大小 | 在线可调，权衡延迟与召回；典型 32~256 |

经验法则：

- **要召回**：先调 `ef_search`（在线、零成本）；
- **要内存效率**：减小 `M`；
- **要构建速度**：减小 `ef_construction`。

### 5.10.2 IVF-PQ 内存优化

| 参数 | 调参建议 |
| --- | --- |
| `nlist` | 经验值 `≈ √N`，N=30 k 取 `173`；过大召回下降明显 |
| `m` | 子向量数量，要求 `dim % m == 0`；典型 `m = dim / 4` |
| `nbits` | 每子向量码本位数，默认 8（256 个码本）；改 4 内存减半但召回急剧下降 |
| `nprobe` | 在线可调；越大越接近精确，但延迟线性增长 |

如需在内存极小的前提下追求高召回，可：

- 先用 IVF-PQ 召回 `top_k * 10` 候选；
- 再用 brute / hnswlib 在候选子集上重排序。

### 5.10.3 Adaptive-HNSW 使用建议

适合 query 难度分布不均的真实业务场景：

- 对**简单 query**（如远离决策边界），首轮就能稳定，避免不必要的 ef 提升；
- 对**困难 query**（如位于密集簇边界），自动重试提升 `ef`，保持高召回；
- `min_ef=32 / max_ef=512` 通常无需调整；`gap_threshold=0.05 / overlap_threshold=0.90` 可按场景微调；
- `oversample`（默认 8）控制多取候选，影响首轮稳定性判定的精度。

### 5.10.4 系统层面调优

- **冷启动加速**：通过 `IndexCache` LRU 缓存避免每次检索重新加载，仅首次有 50~200 ms 冷启动；
- **批量检索**：可在 `backend/scripts/` 中编写脚本一次性提交多个查询，借助 numpy 矢量化提升吞吐；
- **数据存储**：将 `data/` 与 `indexes/` 放在 SSD 上；
- **PostgreSQL**：默认配置已能满足；大规模历史时考虑对 `search_logs(created_at)` 做按月分区。

### 10.1 批量检索（F1）

**使用场景**：一次性提交一组查询（例如 10 个 cell_id），省去逐条 HTTP 往返；同时复用 F2 Redis 检索缓存，重复查询近零成本。适合脚本化批处理与基准评测场景。

**接口**：`POST /api/v1/search/batch`（见 [`backend/app/api/v1/search.py`](../backend/app/api/v1/search.py)）

请求关键字段：

| 字段 | 含义 |
| --- | --- |
| `dataset_id` / `index_id` | 目标数据集与索引（`index_id` 可省略，自动取最新 `ready` 索引）|
| `queries` | 长度 `1~50` 的查询数组，每项 `cell_id` 或 `vector` 二选一 |
| `top_k` / `filters` | 全部查询共享，统一生效 |

**操作步骤**：

1. 登录后获取 JWT：`POST /api/v1/auth/login`；
2. 用 curl / Postman / 脚本提交 JSON body；
3. 响应 `groups[i]` 每条带 `cache_hit` 标记，可判断该子查询是否命中 F2 Redis 缓存。

```bash
curl -X POST http://localhost:8000/api/v1/search/batch \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"dataset_id": 1, "index_id": 3, "top_k": 10, "queries": [
       {"cell_id": "liver_pediatric_0001"}, {"cell_id": "liver_pediatric_0002"}
     ]}'
```

> ⚠️ 前端 SearchPage 当前仅集成 5 个单查询 Tab（by-id / by-vector / multi-dataset / SSE / ensemble），**批量接口暂未在 UI 中暴露**，需通过 API 直调。后续可视情况补充独立的"批处理"页。

### 10.2 SSE 流式检索（F6）

**使用场景**：前端在长 Top-K（≥ 100）或慢后端（brute / IVF-PQ + 大 `nprobe`）场景下，避免"等所有结果出来才一次刷新"，改善等待感。服务端用 Server-Sent Events 协议，每条 hit 单独推送，前端逐条 `append` 到表格。

**接口**：`POST /api/v1/search/by-vector-stream`（见 [`backend/app/api/v1/search.py`](../backend/app/api/v1/search.py)）

**操作步骤**：

1. 进入 `/search` 页面，切到「**SSE 流式**」Tab；
2. 选择数据集与索引（索引可不填，自动选最新 `ready`）；
3. 在「查询向量」文本框粘贴逗号 / 空格分隔的向量（维度需与数据集 `vector_dim` 一致），可叠加可选过滤器；
4. 设置 `Top-K`（默认 10）；
5. 点「**开始流式**」，表格逐条 `append` 一条 hit，顶部计数器实时 +1；
6. 最后一条 `event: done` 触发后，顶部 Descriptions 一次性展示**总耗时 / 返回数量 / 后端 / 距离度量**。

> ⚠️ 浏览器普通 `axios` JSON 请求无法直接吃 SSE 响应，前端使用 fetch + `ReadableStream` + sse-parser 实现，见 [`frontend/src/api/search.ts`](../frontend/src/api/search.ts) 的 `byVectorStream` 方法。
>
> 注：服务端 ANN 仍是一次性算完，SSE 的 ~20 ms 帧间隔是"分帧推流"的视觉效果，**不会增加总 `latency_ms`**——它解决的是感知延迟。

### 10.3 Ensemble 多后端融合（F7）

**使用场景**：同一数据集同时构建多种后端索引（如 `hnswlib` + `faiss-ivfpq` + `brute`），希望"多家投票"提高高难 query 的稳健性。服务端按 z-score 归一化每路距离后再聚合，输出每条结果含 `voted_by`（命中该 cell 的索引 ID 列表），便于分析后端之间的一致性。

**接口**：`POST /api/v1/search/ensemble`（见 [`backend/app/api/v1/search.py`](../backend/app/api/v1/search.py)）

**操作步骤**：

1. 进入 `/search` 页，切到「**Ensemble 多后端**」Tab；
2. 选数据集；
3. 在「参与 ensemble 的索引」多选框中勾选 **2~5 个**同数据集下 `status=ready` 的索引（少于 2 或多于 5 会被前端拦截返回 `400`）；
4. 在「查询细胞 ID」或「查询向量」二选一填写（同时填以 cell_id 优先）；
5. 设置 `Top-K`，可叠加过滤器；
6. 点「**发起检索**」，结果表显示 `rank / cell_id / score（z-score 合并后）/ voted_by / meta`，顶部 Descriptions 显示总耗时与每路索引耗时。

> **经验**：`voted_by` 越长且名次越靠前的结果越值得信赖（多个后端一致命中）；只有单后端命中的结果可能是边界 case，可结合 `score` 决定是否丢弃。
>
> 服务端每路按 `top_k × 3` 取候选再融合，最终严格截断到 `top_k`，避免边界召回损失。

### 10.4 Anthropic Claude RAG（F8）

**使用场景**：在自然语言查询（`/rag`）中切到 Anthropic Claude 后端，适合需要更强中文理解与长 context 推理的场景。v1.1.0 起 RAG 共支持 **4 个 provider**：`mock` / `openai` / `dashscope` / `anthropic`。

**操作步骤**：

1. 在仓库根目录的 `.env` 中调整（参考 [`.env.example`](../.env.example)）：

   ```dotenv
   LLM_PROVIDER=anthropic
   LLM_MODEL=claude-opus-4-20250514
   ANTHROPIC_API_KEY=sk-ant-xxx
   # 也可统一用 LLM_API_KEY；ANTHROPIC_API_KEY 优先级更高
   ```

2. 重启后端容器：`make down && make up`（LLM provider 仅在启动时读取一次）；
3. 浏览器进 `/rag`，选数据集与索引；
4. 在输入框提问，例如：
   - "在儿童肝脏数据中找与 cell_id `liver_pediatric_0001` 类似的 NAFLD 样本 Top-20"；
   - "列出 30 个高表达 ALB 的肝细胞"；
5. 页面同时展示 `parsed`（LLM 解析得到的检索参数）、命中表格与 `answer` 自然语言段落。

> `ANTHROPIC_API_KEY` 优先于 `LLM_API_KEY`，可以单独在生产环境维护一份 Anthropic key 而不影响 dashscope / openai 配置。
> 服务端 LLM 调用实现：[`backend/app/services/rag.py`](../backend/app/services/rag.py)。

5. 「**参数**」展示该 backend 专属参数 JSON（如 hnswlib 的 `M / ef_construction / ef_search`，IVF-PQ 的 `nlist / m / nbits / nprobe`）；
6. 「**最近一次评测**」自动加载该索引最近一次 benchmark 的并发档位表（QPS / P50 / P95 / P99），无评测则降级提示"尚未评测"；
7. 顶部右侧按钮可一键「**去检索**」/「**跑评测**」/「**选用为当前索引**」/「**删除**」。

**接口**：`GET /api/v1/indexes/{id}` + `GET /api/v1/indexes/{id}/latest-benchmark`（见 [`backend/app/api/v1/indexes.py`](../backend/app/api/v1/indexes.py)）。

### 10.8 Redis 检索缓存（F2）

**使用场景**：同一用户对同一索引以**相同 `top_k / filters / 查询`** 重复检索时，无需重新跑 ANN，直接从 Redis 取已落盘结果，平均省 90%+ wall-time。

**命中条件**：服务端按 `(user_id, index_id, top_k, filters, query_hash)` 哈希为 cache key，TTL 默认 5 分钟（可在 [`backend/app/services/search_cache.py`](../backend/app/services/search_cache.py) 调整）。

**何时命中**：

- `by-id` / `by-vector` / `multi-dataset` / `batch` 四个接口自动尝试读写 Redis 缓存；
- `by-vector-stream`（SSE）走的是同一服务层，亦会命中；
- **`ensemble` 与 `exclude_cell_id` 模式暂不缓存**（多路融合无意义；剔除自身的 cell_id 变化频繁）。

**观测命中率**：调 `GET /api/v1/indexes/cache/stats`（见 [`backend/app/api/v1/indexes.py`](../backend/app/api/v1/indexes.py)），响应聚合 IndexCache（进程内 LRU 常驻 ANN 索引）与 SearchCache（Redis 检索结果）**两层指标**：

```json
{
  "capacity": 64, "size": 12, "hits": 530, "misses": 42, "loads": 12,
  "evictions": 0, "hit_ratio": 0.926, "cached_index_ids": [1, 3, 5],
  "search_cache_hits": 218, "search_cache_misses": 47,
  "search_cache_errors": 0, "search_cache_hit_ratio": 0.823
}
```

> **经验**：`search_cache_hit_ratio` 长期低于 30% 多半是查询点分布太散，或 filters 变化频繁——**这是预期行为不必调整**；若反复访问几个固定 cell_id 但仍低，检查 Redis 是否在容器内启动且 `REDIS_URL` 正确。
>
> F1 batch 接口的响应 `groups[i].cache_hit=true` 可在每条查询粒度上直接观测命中状态。

### 10.9 移动端响应式（B2）

**使用场景**：在手机 / 平板上演示或日常巡检，屏幕宽度 `< 768 px`（Ant Design `md` 断点）时，左侧 Sider 自动隐藏 → 顶部出现汉堡按钮 → 抽屉式 Drawer 菜单，不再挤占主区域。

**操作步骤**：

1. 直接用手机浏览器打开站点，或电脑浏览器把窗口宽度拖到 `< 768 px`；
2. Sider 自动隐藏，顶部 Header 左侧出现「☰ 菜单」按钮（`MenuOutlined` 图标）；
3. 点击按钮，左侧弹出 Drawer 抽屉菜单（明色主题），点菜单项即跳路由，跳完自动关闭抽屉；
4. 检索结果表 / 评测图表使用 Ant Design `Col xs/sm/md` 栅格自动堆叠，在窄屏下仍可读；
5. 部分超宽长表（如 admin 用户表）开启横向滚动而非压缩列宽。

**实现细节**：见 [`frontend/src/components/Layout.tsx`](../frontend/src/components/Layout.tsx)，用 `Grid.useBreakpoint()` 监听 `screens.md`，`false` 即视为移动端 → 切换 Drawer。各页面表单 / 表格统一用栅格属性 `xs={24} md={8}` 等做响应式布局。
